## Why

The gallery currently catalogs and displays only image files. Video files (`.mp4`, `.mov`, `.mkv`) placed in cataloged folders are silently ignored by `StorageServiceAdapter.isVideoFile()` (the method does not exist yet) and never reach the database. Users who keep their photos and home videos in the same directory tree have no way to browse, preview, or play their video clips inside the application.

Adding video file support closes this gap. The catalog process will accept the three most common video container formats, generate a representative still-frame thumbnail via FFmpeg so videos appear in the same gallery grid alongside images, and the frontend viewer will switch from `<img>` to a native `<video controls>` element when the selected asset is a video file. Because `video-from-images` (#58) and `video-player` (#66) both depend on FFmpeg being present in the backend container, this change also establishes that runtime dependency in the Dockerfile.

## What Changes

- Extend `StorageServiceAdapter.listFiles()` to include `.mp4`, `.mov`, and `.mkv` extensions alongside the existing image extensions.
- Add a `boolean isVideoFile(String fileName)` helper to `StorageServiceAdapter` and call it from `listFiles()` and `generateThumbnail()`.
- Add `byte[] generateVideoThumbnail(String filePath, int maxWidth, int maxHeight) throws IOException` to `StorageServiceAdapter`: invokes `ffmpeg -i <input> -ss 00:00:01 -vframes 1 -vf scale=<w>:<h> <output>.jpg` via `ProcessBuilder`, reads the resulting JPEG bytes, and deletes the temp file.
- Update `generateThumbnail()` in `StorageServiceAdapter` to delegate to `generateVideoThumbnail()` for video files rather than calling `loadImage()` (which cannot decode video).
- Update `getImageRotation()` in `StorageServiceAdapter` to return `ROTATE_0` immediately for video files without attempting EXIF parsing.
- Update `getExifMetadata()` in `StorageServiceAdapter` to return an all-null `ExifMetadata` record immediately for video files.
- Add `RUN apt-get update && apt-get install -y --no-install-recommends ffmpeg && rm -rf /var/lib/apt/lists/*` to the runtime stage of `JPPhotoManagerWeb/backend/Dockerfile` (the `eclipse-temurin:21-jre-alpine` stage uses `apk`, not `apt-get`; use `RUN apk add --no-cache ffmpeg` instead).
- Add `isVideo: boolean` field to the `Asset` domain model and populate it from `fileName` in `CatalogFolderServiceImpl.createAsset()`.
- Expose `isVideo` in the `AssetDto` HTTP response DTO and the Angular `Asset` model.
- In `GalleryComponent` viewer template, replace the `<img>` with `<video controls>` when `currentViewerAsset.isVideo` is `true`.
- In the thumbnail row template, overlay a play-circle icon on top of the `<img>` when `asset.isVideo` is `true`.
- Add a Cypress component test asserting the play overlay is shown for video assets and that the viewer uses `<video>` instead of `<img>`.

## Capabilities

### New Capabilities

- `video-file-support`: Video files (`.mp4`, `.mov`, `.mkv`) are cataloged, given a still-frame thumbnail generated by FFmpeg, and displayed in the gallery. The viewer renders a native `<video controls>` element for video assets. A play-overlay icon distinguishes video thumbnails from image thumbnails in the list.

### Modified Capabilities

_(none — no existing spec files are affected)_

## Impact

- **`StorageServiceAdapter.java`**: add `isVideoFile()` helper; extend `listFiles()` to include video extensions; delegate `generateThumbnail()` to new `generateVideoThumbnail()` for video files; short-circuit `getImageRotation()` and `getExifMetadata()` for video files.
- **`StoragePort.java`**: no interface change needed — `generateThumbnail`, `getImageRotation`, and `getExifMetadata` already exist; video behavior is an implementation detail of `StorageServiceAdapter`.
- **`CatalogFolderServiceImpl.java`**: populate `asset.setIsVideo(storagePort.isVideoFile(fileName))` when creating an asset.
- **`StoragePort.java`**: add `boolean isVideoFile(String fileName)` method to the interface so `CatalogFolderServiceImpl` can call it without depending on the concrete adapter.
- **`Asset.java`** *(domain model)*: add `boolean isVideo` field.
- **`AssetEntity.java`** *(JPA entity)*: add `boolean is_video` column; covered by a new Flyway migration.
- **`V14__add_asset_is_video.sql`** *(new)*: `ALTER TABLE assets ADD COLUMN is_video BOOLEAN NOT NULL DEFAULT FALSE`.
- **`AssetDto.java`** *(HTTP response DTO)*: add `boolean isVideo` field.
- **`AssetWebMapper.java`** *(MapStruct)*: map `isVideo` from domain model to DTO.
- **`asset.model.ts`**: add `isVideo: boolean` field to the `Asset` interface.
- **`gallery.component.html`**: thumbnail row overlay `<mat-icon>` when `asset.isVideo`; viewer `@if`/`@else` switching `<video controls>` vs `<img>`.
- **`gallery.component.ts`**: helper `isVideo(asset: Asset): boolean` checking extension; no structural changes.
- **`gallery.component.scss`**: `.video-overlay-icon` positioned absolutely over the thumbnail.
- **`gallery.component.cy.ts`**: new tests for play-overlay visibility and viewer `<video>` element.
- **`Dockerfile`** *(backend runtime stage)*: `RUN apk add --no-cache ffmpeg`.
- **No breaking API changes** — `isVideo` is additive to the existing `AssetDto`; all other endpoints are unchanged.
