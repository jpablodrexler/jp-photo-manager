## Context

Refresh tokens are persisted today as a PostgreSQL table (`refresh_tokens`, one row per issued token, FK to `users` with `ON DELETE CASCADE`), mapped through JPA (`RefreshTokenEntity` → `JpaRefreshTokenRepository` → `RefreshTokenRepositoryImpl implements RefreshTokenRepository`). The domain port `RefreshTokenRepository` (`findByToken`, `save`, `deleteByUserId`, `deleteById`) and the domain model `RefreshToken` are already clean hexagonal boundaries — only the infrastructure adapter needs to change. `RefreshTokenServiceImpl` (`infrastructure/service/`) is the sole caller of the port: `issueRefreshToken` creates a new token, `validateAndRotate` reads a token, marks it `revoked = true`, saves it, then issues a replacement, and `revokeAllForUser` deletes every token row for a user.

Redis is already provisioned in this codebase: `redis-distributed-rate-limiting` (#39, already implemented) added the `redis` service to `docker-compose.yml`, the `spring-boot-starter-data-redis` dependency, and `spring.data.redis.host`/`port` properties. That work used a raw Lettuce `RedisClient` for Bucket4j's token-bucket algorithm; this change instead uses Spring Data Redis's autoconfigured `StringRedisTemplate`, which is simpler for plain key/hash/set operations and requires no additional configuration beyond what `spring-boot-starter-data-redis` already autoconfigures from the existing properties.

`session-management` (#46, SDD artifacts created, not yet implemented) plans a `GET /api/auth/sessions` / `DELETE /api/auth/sessions/{id}` pair backed by a numeric token identity. `improvements.md` flags that #46 and #79 should be coordinated so a PostgreSQL `user_agent` column isn't added and immediately discarded. This change does not implement #46; it preserves the `tokenId` (Long) concept via a Redis-native sequence so a future `DELETE /api/auth/sessions/{id}` can still resolve an id to a token without requiring PostgreSQL.

## Goals / Non-Goals

**Goals:**
- Mirror every refresh-token write (issue and rotate) into Redis, alongside the existing PostgreSQL write, without changing the `RefreshTokenRepository` port, the `RefreshToken` domain model, or any use case/controller/DTO that depends on it.
- Design the Redis key scheme (`refresh_token:{token}` hash, `refresh_tokens:user:{userId}` set, `refresh_token:id:{tokenId}` string) so that a later cutover to Redis-only reads and writes requires no key-scheme redesign — only removing the PostgreSQL half of the write and switching `findByToken`/`deleteByUserId`/`deleteById` to read from Redis.
- Preserve numeric `tokenId` semantics (via a Redis `INCR` sequence) so `session-management` (#46)'s planned per-session revoke-by-id endpoint remains implementable without a database identity column.
- Keep PostgreSQL as the sole read source of truth in this phase — a Redis write failure must never fail a login or refresh request.

**Non-Goals:**
- Switching reads to Redis, removing the PostgreSQL write path, or dropping the `refresh_tokens` table — deferred to a follow-up change after a 30-day dual-write observation window (see Migration Plan).
- Implementing `session-management` (#46) itself.
- Introducing Redis Cluster, Sentinel, or any HA topology — the single-node `redis` service from #39 is sufficient; refresh tokens are regenerable (a lost token just forces a fresh login) so there is no data-durability requirement beyond what a single Redis instance provides.
- Changing the rate-limiting Redis usage (`RateLimitFilter`, `LettuceBasedProxyManager`) introduced by #39 — this change uses a separate `StringRedisTemplate` connection and a disjoint key namespace (`refresh_token:*` / `refresh_tokens:*` vs. Bucket4j's own key format), so the two features do not interact.

## Decisions

**Decision: Spring Data Redis (`StringRedisTemplate`) over the existing raw Lettuce `RedisClient` used for rate limiting.**
`spring-boot-starter-data-redis` is already a dependency; Spring Boot autoconfigures a `LettuceConnectionFactory` and `StringRedisTemplate` bean directly from the existing `spring.data.redis.host`/`port` properties, with no additional `@Bean` wiring needed in `AppConfig`. `StringRedisTemplate` gives ergonomic hash (`opsForHash`), set (`opsForSet`), and string (`opsForValue`) operations with built-in `EXPIRE`, which is a better fit for this key/value/TTL shape than Bucket4j's token-bucket-specific `LettuceBasedProxyManager`. Alternative considered: reuse the existing `rateLimitRedisClient`/raw Lettuce connection — rejected, it is wired specifically for Bucket4j's byte-array codec and proxy-manager abstraction, not general key-value access; introducing a second, purpose-built `StringRedisTemplate` is simpler and keeps the two features' Redis usage decoupled.

**Decision: `RedisRefreshTokenStore` as a new infrastructure class encapsulating the key scheme, separate from `RefreshTokenRepositoryImpl`.**
`infrastructure/persistence/redis/RedisRefreshTokenStore.java` owns all Redis key construction and TTL math (`refresh_token:{token}`, `refresh_tokens:user:{userId}`, `refresh_token:id:{tokenId}`, the `INCR` sequence key). `RefreshTokenRepositoryImpl` depends on it and calls it alongside the unchanged JPA calls. This isolates the class that will become the *sole* implementation once Phase 2 cuts over — at that point `RefreshTokenRepositoryImpl` is rewritten to delegate reads to `RedisRefreshTokenStore` instead of `JpaRefreshTokenRepository`, and the store class itself needs no changes. Alternative considered: putting Redis calls directly inline in `RefreshTokenRepositoryImpl` — rejected, it would mix key-scheme details with dual-write orchestration and make the Phase 2 diff larger and riskier.

**Decision: `tokenId` is generated by a Redis `INCR` sequence (`refresh_tokens:token_id_seq`) that mirrors — but is independent of — the PostgreSQL `BIGSERIAL` id.**
The two sequences will not have matching values (Redis's sequence starts fresh at deploy time), which is acceptable because `tokenId` is never compared across stores — it is only ever used to look up a Redis-native token value via `refresh_token:id:{tokenId}`, a purely internal indirection. Alternative considered: reusing the PostgreSQL-generated `tokenId` from the JPA save as the Redis id too — rejected, it would make the Redis write depend on the PostgreSQL write completing first (ordering coupling) for no behavioral benefit, since nothing currently reads `tokenId` cross-store.

**Decision: Revocation is modeled as an immediate Redis `DEL`, not a mirrored "revoked" flag.**
`RefreshTokenServiceImpl.validateAndRotate()` calls `save(existing)` with `revoked = true` set on the in-memory object. In `RefreshTokenRepositoryImpl.save()`, when the incoming domain object has `revoked = true`, the Redis mirror step deletes the token's hash, its `id:{tokenId}` string, and removes it from the user's set — rather than writing a hash with a `revoked=true` field and waiting for TTL expiry. This matches the improvement's stated intent ("revocation is an atomic DEL") and is exactly the behavior Phase 2 needs once Redis becomes the only store: a revoked/rotated-away token must stop resolving via `findByToken` immediately, not linger until TTL. The PostgreSQL row is still soft-revoked (`revoked = true`, kept) unchanged, since Phase 1 does not touch existing PostgreSQL semantics.

**Decision: Redis write failures are caught and logged, never propagated.**
Every Redis mirror call in `RefreshTokenRepositoryImpl` is wrapped so that a `RedisConnectionFailureException` (or any other Redis exception) is caught, logged at `WARN`, and swallowed — the PostgreSQL write (the actual source of truth in this phase) always completes and the HTTP response is unaffected. This mirrors the fail-open posture already established for Redis in `RateLimitFilter` (`redis-distributed-rate-limiting`, #39). Alternative considered: fail the request if Redis is unreachable — rejected, it would make an operationally non-critical mirroring step (whose only purpose is populating Redis ahead of a future cutover) capable of causing login/refresh outages, which is a strictly worse failure mode than temporarily under-populating Redis.

## Risks / Trade-offs

- [Risk] The `refresh_tokens:user:{userId}` Redis set can accumulate stale members for tokens that have already expired out of `refresh_token:{token}` (Redis does not cascade a hash's TTL expiry to sets referencing it). → Mitigation: any code that iterates the set (the future Phase 2 `deleteByUserId`) treats a missing hash as "already gone" and is a no-op for that member; this is the same self-healing pattern used for expiring collection members in Redis generally, and the set itself carries no correctness weight in Phase 1 since PostgreSQL remains authoritative.
- [Risk] Redis and PostgreSQL can drift — a token could exist in PostgreSQL but fail to mirror into Redis (e.g., a transient Redis outage during the write). → Mitigation: this is expected and harmless in Phase 1, since all reads still go through PostgreSQL; Phase 2's cutover verification step (see Migration Plan) explicitly checks for this drift before switching reads, and any token missing from Redis at that point simply forces the affected user to log in again — not a data-loss event, since refresh tokens are regenerable session artifacts, not user data.
- [Risk] The Redis `INCR` sequence and the PostgreSQL `BIGSERIAL` sequence produce different `tokenId` values for the same logical token during Phase 1, which could confuse anyone reading both stores side-by-side for debugging. → Mitigation: documented explicitly in this design; `tokenId` is never compared cross-store by any code path, only used as an internal Redis-side indirection.
- [Risk] Introducing a second `StringRedisTemplate`-based Redis usage alongside the existing Bucket4j/Lettuce rate-limit usage increases the connection pool footprint against the single-node `redis` service. → Mitigation: `StringRedisTemplate`'s autoconfigured `LettuceConnectionFactory` reuses Lettuce's shared, thread-safe connection multiplexing (a single connection handles concurrent commands), so the additional footprint is minimal; no dedicated connection pool sizing change is required for the `redis:7-alpine` container's default resource limits already set in `docker-compose.yml`.

## Migration Plan

1. **Deploy 1 (this change — dual-write groundwork)**: Ship `RedisRefreshTokenStore` and the updated `RefreshTokenRepositoryImpl` that mirrors every `save()` (issue and revoke-via-delete) into Redis alongside the existing PostgreSQL write. Reads (`findByToken`) continue to come from PostgreSQL exclusively; Redis is populated but not yet consulted. From this point forward, every token issued or rotated exists in both stores; tokens issued before this deploy exist only in PostgreSQL until they are naturally rotated (on their next refresh) or expire.
2. **Observation window (30 days)**: One full refresh-token lifetime elapses, by which point every currently-active token has been rotated at least once since Deploy 1 and therefore exists in Redis. No code change during this window.
3. **Verification**: An operator (or a post-window smoke check) confirms that active, non-expired PostgreSQL `refresh_tokens` rows have a corresponding Redis `refresh_token:{token}` hash, and spot-checks that `POST /api/auth/refresh` behaves identically whether validated against PostgreSQL or Redis for a sample of tokens.
4. **Deploy 2 (follow-up change, out of scope here)**: Switch `findByToken`/`deleteByUserId`/`deleteById` to read from `RedisRefreshTokenStore` only; stop writing to PostgreSQL; apply a new Flyway migration (V32 or later) dropping the `refresh_tokens` table and its FK to `users`. `RefreshTokenRepositoryImpl` is rewritten to depend solely on `RedisRefreshTokenStore`; `JpaRefreshTokenRepository`, `RefreshTokenEntity`, and `RefreshTokenEntityMapper` are deleted.
5. **Rollback strategy**: Between Deploy 1 and Deploy 2, rollback is trivial — revert to the previous release, which still reads/writes PostgreSQL exclusively (Redis mirroring was additive and inert for reads). After Deploy 2 (table dropped), rollback requires restoring the table from a database backup; this is the same class of risk already accepted for the `mongodb-exif-store` (#72) `asset_exif` schema drop.

## Open Questions

- Should the Redis mirror TTL be set to the exact remaining PostgreSQL `expires_at` (as designed here), or a fixed 30 days from mirror-write time? This design uses the exact remaining lifetime so a token rotated on day 25 of its PostgreSQL life gets a ~5-day Redis TTL, keeping the two stores' expiry semantics identical — recommended, no open concern, noted here for the Phase 2 implementer's awareness.
- Should Deploy 2's verification step be a manual operator check (as it was for `mongodb-exif-store`) or an automated startup consistency check? Deferred to the follow-up change's own design; either is compatible with the key scheme established here.
