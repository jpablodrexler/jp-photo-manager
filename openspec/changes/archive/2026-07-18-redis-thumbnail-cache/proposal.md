## Why

`ThumbnailStorageServiceAdapter` (the sole implementation of `ThumbnailPort`) reads and writes thumbnails as `{assetId}.bin` files on local disk. In a load-balanced deployment with multiple backend instances behind a shared PostgreSQL database, each instance has its own filesystem — a thumbnail generated by the instance that handled cataloging is not visible to a request routed to a different instance, forcing a redundant regeneration (or a shared network filesystem dependency that the current design doesn't require anywhere else). A shared, server-side thumbnail cache removes that coupling and cuts disk I/O for the hottest read path in the gallery (every thumbnail grid cell requests one). Redis is already provisioned and in production use for rate limiting (`redis-distributed-rate-limiting`, #78) and refresh-token mirroring (`redis-refresh-tokens`, #79), so no new infrastructure is required.

## What Changes

- Add a Redis-backed L2 cache in front of the existing disk store inside `ThumbnailStorageServiceAdapter`:
  - `loadThumbnail(blobName)`: try `GET asset:thumbnail:{assetId}` first; on a hit, return the cached bytes with no disk I/O; on a miss, read from disk as today, populate the cache via `SETEX` with a 24-hour TTL, and return.
  - `saveThumbnail(blobName, data)`: write to disk as today, then also `SETEX asset:thumbnail:{assetId}` with the same 24-hour TTL so a freshly generated thumbnail is immediately servable from cache.
  - `deleteThumbnail(blobName)`: delete the disk file as today, then `DEL asset:thumbnail:{assetId}` so a stale cache entry is never served after removal. Because this is the single method all three delete-path use cases (`DeleteAssetsUseCaseImpl`, `PurgeAssetsUseCaseImpl`, `PruneDeletedFoldersUseCaseImpl`) already call, no use-case-level changes are required.
- Configure Redis's `allkeys-lru` eviction policy (already the policy used by the existing Redis deployment) so popular thumbnails are retained and cold ones are evicted automatically under memory pressure.
- Add a dedicated `RedisTemplate<String, byte[]>` bean (raw byte serialization, no JDK object serialization) so thumbnail bytes are stored and retrieved without added serialization overhead.
- All Redis operations are wrapped so a Redis outage degrades to today's disk-only behavior (never fails a thumbnail request) and is logged at `WARN`, mirroring the existing pattern in `RedisRefreshTokenStore`.
- No changes to the `ThumbnailPort` interface, its method signatures, or any of its callers — the cache is entirely internal to the adapter.

## Capabilities

### New Capabilities
- `redis-thumbnail-cache`: a Redis-backed L2 cache for thumbnail bytes sitting in front of the disk-backed `ThumbnailStorageServiceAdapter`, with a 24-hour TTL, `allkeys-lru` eviction, and cache invalidation on thumbnail deletion.

### Modified Capabilities
(none — `thumbnail-http-cache` (#26, browser `Cache-Control` header) and `server-side-spring-cache` (#28, Caffeine-backed `@Cacheable` for home stats/sub-folders/EXIF) keep their existing requirements unchanged; this change adds an independent server-side tier without altering either.)

## Impact

- **Affected code:** `infrastructure/service/ThumbnailStorageServiceAdapter.java` (the only class that changes behavior); a new `RedisTemplate<String, byte[]>` bean in `config/AppConfig.java` (or a new `config/RedisThumbnailCacheConfig.java`); new configuration properties in `application.yml` (TTL, enable/disable flag).
- **Dependencies:** none new — `spring-boot-starter-data-redis` is already a Maven dependency (used by `RedisRefreshTokenStore`), and the `redis` service is already provisioned in `docker-compose.yml` / Kubernetes manifests by `redis-distributed-rate-limiting` (#78).
- **Callers unaffected:** `GetAssetThumbnailUseCaseImpl`, `DeleteAssetsUseCaseImpl`, `PurgeAssetsUseCaseImpl`, and `PruneDeletedFoldersUseCaseImpl` all call the existing `ThumbnailPort` methods and require no code changes.
- **Operational:** Redis memory usage grows with the number of distinct thumbnails served within the TTL window; `allkeys-lru` bounds this automatically. If thumbnail caching is found to contend with rate-limit buckets or refresh-token keys on the same Redis instance, a separate keyspace/maxmemory policy or a dedicated Redis instance can be introduced later without any application-code change (only connection configuration).
