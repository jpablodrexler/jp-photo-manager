- [x] 3.1 Rewrite `infrastructure/persistence/adapter/AssetExifRepositoryImpl.java` to implement `AssetExifRepository` (`findByAssetId`, `save`, `deleteByAssetId`) using `JpaAssetExifRepository`/`AssetExifEntityMapper` instead of `MongoAssetExifRepository`/`AssetExifDocumentMapper` — **Fixed:** restored from `e336657`, with two corrections: (1) added `entity.setRawExif(assetExif.getRawExif())` to `save()`, which the original historical version was missing (a pre-existing gap — the current `AssetExifProcessor`/`CatalogFolderServiceAdapter` callers populate `rawExif` on the domain object before calling `save()`, so the adapter must persist it); (2) added an explicit `entityManager.flush()` after `jpa.save(entity)` — **critical correctness bug found via `UploadKafkaPipelineIntegrationTest` and confirmed with Hibernate SQL logging**: `AssetExifProcessor.onAssetUploaded()` first calls the readOnly-transactional `findByAssetId()` (participating in the same physical transaction, since `AssetExifRepository`'s query methods inherit `SimpleJpaRepository`'s class-level `@Transactional(readOnly = true)`), which leaves the Hibernate session's flush mode set to `MANUAL` for the rest of that transaction; `MANUAL` means Hibernate does not auto-flush even at commit, so the subsequent `save()`'s insert/update was silently never written to `asset_exif` — the transaction still committed cleanly (other writes on `assets`, made via bulk `@Modifying` queries that bypass the persistence context, succeeded normally), masking the loss entirely: no exception, no rollback, just a missing row. ## 9. Unrelated integration test fixes (requested follow-up, not part of the original revert scope)

- [x] 9.1 `TagIntegrationTest.removeTag_notAssigned_throws` — split into `removeTag_tagDoesNotExist_throwsTagNotFoundException` and `removeTag_tagExistsButNotAssignedToThisAsset_throwsNoSuchElementException`; the original test used a tag name that was never created at all, which throws `TagNotFoundException` (from `tagRepository.findByName(...).orElseThrow(...)`) rather than the `NoSuchElementException` the test expected (that exception is a separate branch, reached only when the tag exists but isn't assigned to the given asset).
- [x] 9.2 `CatalogBatchIntegrationTest`, `UploadKafkaPipelineIntegrationTest`, `UploadAssetIntegrationTest`, `ConvertKafkaPipelineIntegrationTest`, `SyncKafkaPipelineIntegrationTest` — added `@WithMockUser(roles = "ADMIN")` (class- or method-level); all five call `@PreAuthorize("hasRole('ADMIN')")`-protected use cases without ever authenticating a test principal.
- [x] 9.3 `ConvertAssetsUseCaseImpl.execute()` / `SyncAssetsUseCaseImpl.execute()` — **production bug, not just a test gap**: both combine `@Async` with `@PreAuthorize` on the same method; the security check runs on the `@Async` executor's worker thread, which does not inherit the calling thread's `SecurityContext` by default — so in production too, every authenticated call to `/api/convert/run` or `/api/sync/run` would silently fail the auth check on that background thread (the exception is swallowed inside the unobserved returned `CompletableFuture`, so nothing is ever logged) and the operation would hang forever waiting for a `done` Kafka message that's never sent. Fixed by wrapping the `taskExecutor` bean in `AppConfig` with Spring Security's `DelegatingSecurityContextAsyncTaskExecutor` (propagates the submitting thread's `SecurityContext` to the worker thread for the task's duration) and switching both `@Async` annotations to the explicit `@Async("taskExecutor")` qualifier — `@Async`'s default by-name executor lookup is not reliable once more than one `TaskExecutor`-typed bean exists (e.g. Spring Boot's auto-configured `applicationTaskExecutor`), which is why the qualifier is needed in addition to the executor wrapper.
- [x] 9.4 `UploadKafkaPipelineIntegrationTest` — separate, second pre-existing bug uncovered once 9.2's auth fix let execution reach further into the test: both test methods reused the same static `catalogFolder` path with no per-test cleanup, so whichever method ran second found two `Folder` rows with an identical path and `folderRepository.findByPath(...)` (a single-result query) threw `IncorrectResultSizeDataAccessException`. Fixed by giving each test its own unique folder subpath (matching the per-test-subfolder pattern already used in `ExifMetadataIntegrationTest`).
